# Persistencia con SQL

## Instalación de motores de bases de datos SQL

Existen múltiples sistemas de gestión de bases de datos relacionales (RDBMS) ampliamente utilizados. Entre ellos destacan **MariaDB**, **PostgreSQL** y **SQL Server**, cada uno con sus particularidades y métodos de instalación.

### MariaDB

MariaDB es un fork de MySQL, diseñado para ser completamente compatible y ofrecer mejoras en rendimiento y características.

**Instalación en Linux (Debian/Ubuntu):**

```bash
sudo apt update
sudo apt install mariadb-server mariadb-client
sudo systemctl start mariadb
sudo systemctl enable mariadb
```

**Instalación en Windows:**

1. Descargar el instalador desde la página oficial de MariaDB.
2. Ejecutar el instalador y seguir los pasos, seleccionando la configuración típica.
3. Configurar la contraseña del usuario root y los puertos por defecto (3306).

### PostgreSQL

PostgreSQL es conocido por su robustez y soporte avanzado para operaciones complejas y tipos de datos.

**Instalación en Linux (Debian/Ubuntu):**

```bash
sudo apt update
sudo apt install postgresql postgresql-contrib
sudo systemctl start postgresql
sudo systemctl enable postgresql
```

**Instalación en Windows:**

1. Descargar el instalador desde el sitio oficial de PostgreSQL.
2. Seguir los pasos de instalación, definiendo contraseña para el usuario `postgres`.

### SQL Server

SQL Server, desarrollado por Microsoft, es un RDBMS corporativo que ofrece integración profunda con entornos Windows y herramientas de BI.

**Instalación en Linux:**

```bash
sudo su
curl https://packages.microsoft.com/keys/microsoft.asc | apt-key add -
curl https://packages.microsoft.com/config/ubuntu/20.04/mssql-server-2022.list | tee /etc/apt/sources.list.d/mssql-server.list
sudo apt update
sudo apt install -y mssql-server
sudo /opt/mssql/bin/mssql-conf setup
sudo systemctl status mssql-server
```

En este apunte, se utilizará **MariaDB** para los ejemplos prácticos, aunque los conceptos son fácilmente adaptables a otros RDBMS.

## Configuración básica de MariaDB

Una vez instalado el servidor, se requiere asegurar la conectividad y definir los parámetros iniciales:

* **Usuario y contraseña:** Se recomienda crear un usuario específico para la aplicación en lugar de utilizar `root`.
* **Base de datos de prueba:** Para iniciar se puede crear:

```sql
CREATE DATABASE universidad;
USE universidad;
CREATE TABLE estudiantes (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(50),
    apellido VARCHAR(50),
    email VARCHAR(100)
);
```

* **Permisos:** Asignar permisos adecuados al usuario de la aplicación:

```sql
CREATE USER 'appuser'@'localhost' IDENTIFIED BY 'password123';
GRANT ALL PRIVILEGES ON universidad.* TO 'appuser'@'localhost';
FLUSH PRIVILEGES;
```

* **Puerto y host:** MariaDB, por defecto, escucha en el puerto 3306 y en `localhost`. Para conexiones remotas, se debe configurar el archivo `my.cnf` o `50-server.cnf`.

## Conexión con driver nativo (sin ORM)

En aplicaciones Node.js, se puede establecer una conexión directa a MariaDB mediante el driver oficial `mysql2`. Este enfoque permite control total sobre consultas y transacciones, sin la abstracción que ofrece un ORM.

**Instalación del driver:**

```bash
npm install mysql2
```

**Conexión simple:**

```javascript
const mysql = require('mysql2');

const connection = mysql.createConnection({
  host: 'localhost',
  user: 'appuser',
  password: 'password123',
  database: 'universidad'
});

connection.connect(error => {
  if (error) {
    console.error('Error al conectar a la base de datos:', error);
  } else {
    console.log('Conexión establecida con éxito.');
  }
});

// Ejemplo de inserción
const estudiante = { nombre: 'Sebastián', apellido: 'Pérez', email: 'sebastian@ejemplo.com' };
connection.query('INSERT INTO estudiantes SET ?', estudiante, (error, results) => {
  if (error) throw error;
  console.log('Estudiante insertado, ID:', results.insertId);
});

connection.end();
```

En este ejemplo, se demuestra la inserción de datos de manera directa, utilizando parámetros seguros (`?`) para evitar inyecciones SQL.

## Pool de conexiones

El uso de un **pool de conexiones** es una práctica recomendada en aplicaciones de producción. Un pool mantiene un conjunto de conexiones abiertas, reutilizables, evitando la sobrecarga de abrir y cerrar conexiones continuamente.

**Configuración de un pool con `mysql2`:**

```javascript
const pool = mysql.createPool({
  host: 'localhost',
  user: 'appuser',
  password: 'password123',
  database: 'universidad',
  waitForConnections: true,
  connectionLimit: 10,
  queueLimit: 0
});

// Uso del pool
pool.query('SELECT * FROM estudiantes', (error, results) => {
  if (error) throw error;
  console.log('Lista de estudiantes:', results);
});
```

El pool maneja automáticamente la gestión de conexiones, optimizando el rendimiento bajo carga y reduciendo la latencia en consultas frecuentes.

## Validación de inserción con Postman

Para aplicaciones web que exponen endpoints para manipular datos, es importante verificar que las inserciones se reflejen correctamente en la base de datos. **Postman** es una herramienta que permite realizar peticiones HTTP a APIs de manera sencilla y validar los resultados.

**Procedimiento:**

1. Crear un endpoint en la aplicación Node.js que acepte un POST con los datos de un estudiante.
2. En Postman, configurar una solicitud POST al endpoint correspondiente, enviando un JSON:

```json
{
  "nombre": "Laura",
  "apellido": "Gómez",
  "email": "laura@ejemplo.com"
}
```

3. Ejecutar la solicitud y verificar la respuesta, que debería incluir un mensaje de éxito y el `id` generado.
4. Consultar directamente en MariaDB:

```sql
SELECT * FROM estudiantes WHERE email = 'laura@ejemplo.com';
```

Este procedimiento asegura que la inserción no solo es aceptada por la API, sino que efectivamente se persiste en la base de datos, cerrando el ciclo de validación de datos.

# Persistencia con SQL: consultas avanzadas, transacciones y manejo de errores

El trabajo con bases de datos relacionales va más allá de simples inserciones y consultas. Para aplicaciones confiables y eficientes, es necesario comprender cómo realizar consultas complejas, garantizar la integridad mediante transacciones y manejar errores de forma robusta.

## Consultas avanzadas

Las consultas SQL permiten recuperar, filtrar y combinar datos de múltiples maneras. La comprensión de operaciones avanzadas como `JOIN`, `WHERE`, `ORDER BY`, `GROUP BY` y subconsultas es crucial.

### Filtrado y ordenamiento

Se puede obtener un subconjunto de estudiantes según criterios específicos y ordenarlos por campos determinados:

```sql
SELECT nombre, apellido, email
FROM estudiantes
WHERE apellido LIKE 'G%'
ORDER BY nombre ASC;
```

Este ejemplo filtra todos los estudiantes cuyo apellido comienza con la letra "G" y los ordena alfabéticamente por nombre.

### Consultas con JOIN

Cuando se tiene más de una tabla relacionada, `JOIN` permite combinar registros basados en claves externas:

```sql
CREATE TABLE cursos (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(50)
);

CREATE TABLE matriculas (
    id INT AUTO_INCREMENT PRIMARY KEY,
    estudiante_id INT,
    curso_id INT,
    FOREIGN KEY (estudiante_id) REFERENCES estudiantes(id),
    FOREIGN KEY (curso_id) REFERENCES cursos(id)
);

-- Consulta de todos los estudiantes con sus cursos
SELECT e.nombre, e.apellido, c.nombre AS curso
FROM estudiantes e
JOIN matriculas m ON e.id = m.estudiante_id
JOIN cursos c ON m.curso_id = c.id;
```

El uso de `JOIN` asegura que se obtenga información combinada de distintas tablas respetando las relaciones definidas.

### Subconsultas y agregaciones

Se pueden realizar cálculos o filtrar resultados utilizando subconsultas o funciones agregadas:

```sql
-- Contar estudiantes por curso
SELECT c.nombre AS curso, COUNT(m.estudiante_id) AS total_estudiantes
FROM cursos c
LEFT JOIN matriculas m ON c.id = m.curso_id
GROUP BY c.id;
```

Este tipo de consultas permite obtener métricas agregadas sin necesidad de procesarlas en la aplicación.

## Transacciones

Las transacciones garantizan que un conjunto de operaciones SQL se ejecute de manera atómica: o todas se completan correctamente o ninguna se aplica, asegurando la integridad de los datos.

### Uso de transacciones en Node.js con MariaDB

```javascript id="d9p5v2"
const mysql = require('mysql2/promise');

async function insertarEstudianteYMatricula(estudiante, cursoId) {
  const connection = await mysql.createConnection({
    host: 'localhost',
    user: 'appuser',
    password: 'password123',
    database: 'universidad'
  });

  try {
    await connection.beginTransaction();

    const [resultEstudiante] = await connection.query(
      'INSERT INTO estudiantes (nombre, apellido, email) VALUES (?, ?, ?)',
      [estudiante.nombre, estudiante.apellido, estudiante.email]
    );

    const estudianteId = resultEstudiante.insertId;

    await connection.query(
      'INSERT INTO matriculas (estudiante_id, curso_id) VALUES (?, ?)',
      [estudianteId, cursoId]
    );

    await connection.commit();
    console.log('Transacción completada con éxito.');
  } catch (error) {
    await connection.rollback();
    console.error('Error en la transacción, se revertieron los cambios:', error);
  } finally {
    await connection.end();
  }
}

insertarEstudianteYMatricula(
  { nombre: 'María', apellido: 'López', email: 'maria@ejemplo.com' },
  1
);
```

En este ejemplo, si la segunda inserción falla, la primera también se revierte, evitando inconsistencias en la base de datos.

## Manejo de errores

El manejo adecuado de errores permite garantizar estabilidad y seguridad. Existen diferentes tipos de errores que deben considerarse:

* **Errores de conexión:** ocurren cuando la base de datos no está disponible o las credenciales son incorrectas.
* **Errores de sintaxis SQL:** consultas mal formadas.
* **Errores de integridad:** violación de restricciones, como claves primarias duplicadas o violaciones de `FOREIGN KEY`.

### Estrategias de manejo

1. **Captura y logueo:** registrar el error para diagnóstico y auditoría.
2. **Rollback en transacciones:** asegurar que operaciones incompletas no afecten la base de datos.
3. **Respuestas claras en la API:** devolver mensajes consistentes a los consumidores, sin exponer detalles sensibles.

```javascript id="f7w8k3"
pool.query('INSERT INTO estudiantes (nombre, apellido, email) VALUES (?, ?, ?)', 
  ['Ana', 'Martínez', 'ana@ejemplo.com'], 
  (error, results) => {
    if (error) {
      if (error.code === 'ER_DUP_ENTRY') {
        console.log('Error: ya existe un estudiante con ese email.');
      } else {
        console.error('Error desconocido:', error);
      }
    } else {
      console.log('Estudiante insertado con ID:', results.insertId);
    }
});
```

Este enfoque permite diferenciar errores esperados (como duplicados) de errores inesperados, facilitando la depuración y la experiencia de usuario.

## Buenas prácticas

* Siempre parametrizar consultas para prevenir **inyecciones SQL**.
* Evitar conexiones abiertas innecesarias, utilizando **pools de conexiones**.
* Validar los datos antes de insertarlos en la base de datos.
* Documentar la estructura de tablas y relaciones para mantener claridad en proyectos grandes.
* Usar transacciones cuando varias operaciones dependen unas de otras.
